Introduction1.1 PurposeThis Technical Requirements Document (TRD) defines the architecture, technology choices, and engineering rationale behind Hisab Kitab — a digital khatabook/ledger application for shopkeepers. It is the technical companion to the Product Requirements Document (PRD) and translates product requirements into concrete system design decisions, with the reasoning behind each one recorded so future contributors understand not just what was built, but why.1.2 Problem StatementKhatabook wants a digital ledger where shopkeepers add transactions and see a running balance update immediately. Transaction history must be paginated, every transaction must be editable/deletable with a complete audit trail, and multiple shopkeepers must never be able to edit the same transaction simultaneously.1.3 ScopeThis document covers the core ledger system, real-time synchronization, concurrency and data-integrity controls, the extended Worker & Salary and Inventory & Loss modules, AI-assisted paper-ledger migration, and the engineering conventions that govern how the codebase is organized and how the team collaborates on it.1.4 AudienceBackend and frontend engineers implementing the systemReviewers and future contributors onboarding onto the codebaseProject stakeholders validating technical feasibility of product requirements2. It was chosen over component-library CSS-in-JS or hand-written stylesheets for a few reasons specific to this project:Speed for a small team — utility classes let any of the three engineers style a component without switching files or agreeing on a separate naming convention (BEM, CSS Modules, etc.).Consistency by default — a shared Tailwind config (spacing scale, color palette, typography) keeps the UI visually consistent across features built by different people, which matters because the team is organized by feature folder (contacts/, workers/, inventory/) rather than by a single frontend owner.No runtime CSS-in-JS cost — Tailwind compiles to static CSS at build time, which keeps the app lightweight — important given shopkeepers are often on lower-end phones and patchy networks (see the PWA/offline requirement).Pairs naturally with shadcn/ui-style primitives — the components/ui/ folder (dumb, reusable primitives) is built directly on Tailwind utility classes, so there is no styling abstraction layer to maintain on top of it.Tailwind's config (tailwind.config.ts) lives at the project root and defines the shared design tokens — colors, spacing, font sizes — referenced by every component under src/components/.3. System Architecture OverviewHisab Kitab is a single Next.js application with a strict internal boundary between presentation and business logic, rather than a split into separate frontend/backend repositories. The boundary is enforced by import direction, not by physical separation:components/ (client, presentation-only) never imports from server/ directly.API routes under app/api/ are the only bridge between the browser and server/ business logic.server/ code never imports React and contains no JSX.Requests flow: Browser → Next.js App Router route (thin) → server/services (business logic: locking, version checks, rollup updates) → server/repositories (Prisma data access) → PostgreSQL. Real-time updates flow the opposite direction: PostgreSQL NOTIFY → server/realtime/pg-listener.ts → open SSE connections → browser.3.1 Why a Single Codebase, Not Separate Frontend/Backend ReposSplitting Next.js into literal frontend/ and backend/ top-level projects would fight the framework itself — routing, server components, and bundling all assume a unified project. Instead, the frontend/backend boundary is enforced by convention (import direction) and reviewed in PRs, which keeps deployment simple (one Cloud Run service) while still preventing business logic from leaking into client components.3.2 Request LifecycleThin routes: app/api/*/route.ts files only parse the request, call a server/services/* function, and return the response.Fat services: all logic — locking, optimistic version checks, rollup updates — lives in server/services/, independently testable without HTTP.Repository isolation: server/repositories/ is the only place Prisma calls happen for core entities, so schema changes have a contained blast radius.Shared validation: one Zod schema per resource in server/validators/, used by both the API route and the corresponding client form — server and form can never drift apart on what's valid.4. Core Architectural DecisionsThis section records the significant design decisions for the core ledger system, the alternatives considered, and why each final choice was made.4.1 Concurrency Control — Preventing Simultaneous EditsTwo standard approaches exist for preventing two shopkeepers from editing the same transaction at once:ApproachHow It WorksTrade-offOptimistic lockingAnyone can open the edit form at any time. Whoever saves first wins; the second person is rejected only at save time.Cheap, no live signaling — but failure is discovered late, after the user has already done the work of editing.Pessimistic lockingThe moment someone opens a transaction to edit, it is locked for everyone else in real time.Feels live and prevents wasted effort, but needs a mechanism to release stale locks (crashed tabs, closed browsers).▶  DecisionUse both, layered. A pessimistic soft-lock (lockedBy + lockedAt, ~2-minute TTL, refreshed by a client heartbeat, auto-released on close/disconnect/timeout) is the primary, live-feeling mechanism.An optimistic version integer underneath acts as a safety net — every write is rejected if the version has changed since it was read, guaranteeing no silent overwrite even if a lock goes stale.This hybrid was chosen because the same real-time channel used for balance updates (4.2) can broadcast lock status at near-zero extra cost.4.2 Real-Time Balance UpdatesStandard web requests are one-directional — the browser asks, the server answers. If Shopkeeper A adds a transaction, Shopkeeper B (viewing the same contact's ledger) needs their balance to update without any action on their part, which requires the server to push into B's already-open tab.OptionDescriptionVerdictSSE (Server-Sent Events)Postgres write → NOTIFY → API route LISTEN-ing → pushed down an open SSE connection. One-directional (server→browser), built entirely on Postgres.SelectedManaged service (Pusher / Ably)Third party delivers the message instead of the app.Rejected — external paid dependency, another moving partSelf-hosted Socket.ioFull two-directional real-time.Rejected — more power than needed; requires an always-running server process, awkward alongside serverless Cloud Run▶  DecisionSSE via Postgres LISTEN/NOTIFY. The app only ever needs server→browser pushes (balance changed, lock acquired/released) — never browser-to-browser messaging — so SSE covers the requirement completely with zero new infrastructure and fits Cloud Run's serverless model.4.3 AuthenticationReal authentication is implemented from day one rather than a stubbed fake user. Multi-shopkeeper editing, locking, and the audit trail all require knowing who is doing what from the very first feature built — retrofitting identity later would mean redoing all three.4.4 Ledger Model — Contact-Based, Not a Single Shared BalanceTwo models were considered: a single shop-wide cash-book (one running total shared by all shopkeepers, like a physical register on the counter) versus per-person ledgers.▶  DecisionPer-person (“Contact”) model. The product requires showing total dues, dues for a specific person, and what the shop owes to vendors — a single shop-wide number cannot answer “how much does Ramesh owe me,” the single most common question a real Khatabook user asks.Direction is derived, not stored: rather than separate Customer and Vendor tables, one Contact entity represents any person or business, because the underlying math is identical — only the sign flips.Contact = a customer → they owe the shop → a dueContact = a vendor → the shop owes them → a payableEach transaction is either “You Gave” (increases what the contact owes) or “You Got” (decreases it). Running balance = sum(You Gave) − sum(You Got). Positive = they owe you (due); negative = you owe them (payable). This single model yields total dues, total payables, per-person history, and full transaction history from one table.4.5 Deletes — Soft Delete OnlyNothing is ever hard-deleted. isDeleted, deletedAt, and deletedBy fields are set instead. This single mechanism serves two requirements at once: the audit trail (the “before” state must exist to log what changed) and the Trash/Restore feature (owners can undo an accidental delete within a 7-day grace period).4.6 Money Storage — Decimal, Not Float or IntCurrency math performed with floating-point numbers accumulates rounding errors (the classic 0.1 + 0.2 !== 0.3 problem). All monetary amounts are stored as Postgres Decimal types, which represent currency exactly.4.7 Balance Consistency — Preventing Race ConditionsThis is distinct from the edit-locking in 4.1, which stops two people editing the same existing transaction. Two shopkeepers can also each add a different new transaction for the same contact at nearly the same instant — if both read the “current balance” before either writes, one update can silently overwrite the other.▶  DecisionEvery write that touches a contact's balance runs inside a single Postgres transaction using SELECT ... FOR UPDATE to row-lock that Contact row. The second writer is physically forced to wait until the first commits, then reads the already-updated balance before applying its own change — no lost updates, guaranteed by the database itself rather than application logic.4.8 Pagination at ScaleWith thousands of transactions per ledger, standard offset/limit pagination (“skip 5000, take 50”) gets slower the deeper a user pages, because Postgres must scan and discard every skipped row.▶  DecisionCursor-based pagination, keyed on (createdAt, id) with a composite index — each page is a fast indexed range lookup regardless of depth.A balanceAfter snapshot is cached on every transaction row so the UI never has to recompute a running total for a page; it just reads the stored value.4.9 Daily / Monthly EarningsBecause Hisab Kitab is built to scale to thousands of users from day one, daily and monthly totals are backed by a cached EarningsRollup table from the start rather than deferring aggregation performance work until it becomes a problem.One row per shop per day: (shopId, date, totalEarned, totalLoss).Updated incrementally inside the same database transaction as every write that affects it (new transaction, edit, delete/restore, stock loss) — the rollup and underlying data can never drift apart.The dashboard reads directly from EarningsRollup — a single indexed row lookup — instead of re-aggregating transaction history on every load, keeping dashboard load times flat regardless of transaction volume.Monthly totals are derived by summing daily rows for the month rather than maintaining a separate monthly rollup, since 28–31 indexed row reads is effectively free.5. MVP Feature Set5.1 Core Features (from the Original Problem Statement)Add transactions per contact — every add goes through the race-condition-safe balance update (4.7).Real-time running balance — via SSE (4.2).Paginated transaction history — cursor-based (4.8).Edit/delete with audit trail — soft delete (4.5) plus a TransactionAudit log, written in the same database transaction as the change, storing who changed it, when, and both old and new values as JSON snapshots.No simultaneous edits — hybrid locking (4.1).5.2 Additional MVP-Scope FeaturesRole-based permissions (Owner/Staff) — e.g. only the Owner can delete a transaction or view the full audit trail.Due date + reminders per contact — dashboard surfaces overdue contacts.Search & filter transactions — by contact, date range, type, amount.Receipt/photo attachment — stored via a GCS bucket URL on the transaction.Trash/restore — a “Recently Deleted” screen where an Owner can restore an accidental delete within 7 days.Shareable ledger link / PDF export — a read-only link or PDF of one contact's ledger.Multi-shop support — a user can belong to more than one shop via a ShopMember join table.Analytics dashboard — money in vs out over time, top contacts by due amount, monthly trend.Opening balance / bulk import — set an opening balance per contact, or bulk-import via CSV.In-app notifications — alerts when a teammate edits/deletes a transaction, plus due-date reminders.Hindi/regional language toggle — built in from the start since i18n is painful to retrofit later.PWA / installable, offline-aware — works offline for viewing, queues writes until back online.Daily & monthly earnings view — see 4.9.6. Extended ModulesEach extended module deliberately reuses the patterns already established for the core ledger — locking, versioning, audit trail, soft delete — rather than inventing a parallel set of rules per feature area.6.1 Worker & Salary ManagementEntitiesWorker — belongs to a Shop; payType is MONTHLY or DAILY_WAGE; stores monthlySalary or dailyWageRate accordingly, plus designation, joiningDate, active status.Attendance — one row per worker per day: PRESENT / ABSENT / HALF_DAY / LEAVE, who marked it, when.SalaryPayment — amount, periodStart/periodEnd, paidOn, paidBy, note, status (PAID/PENDING) — carries the same version / lockedBy / lockedAt fields as Transaction (4.1).SalaryAudit — the same old-value/new-value/who/when pattern as TransactionAudit.Pending Salary CalculationMONTHLY workers: monthlySalary × full months since joining/last payment − total paid.DAILY_WAGE workers: (PRESENT days × dailyWageRate) + (HALF_DAY days × dailyWageRate ÷ 2) − total paid, computed from the Attendance log.This surfaces on the dashboard as a “Salary Payable” figure alongside the dues/payables already coming from contacts (4.4), so an owner sees everything the shop owes, to anyone, in one place.6.2 Inventory & Loss TrackingEntitiesProduct — belongs to a Shop: name, SKU, category, unit, purchasePrice, sellingPrice, reorderLevel.ProductBatch — a specific restock batch: batchNumber, quantity, expiryDate, purchaseDate, purchasePrice.StockEntry — every stock movement (PURCHASE, SALE, ADJUSTMENT, LOSS). PURCHASE/SALE auto-approve; ADJUSTMENT/LOSS require Owner approval before they count.Product.currentStock — a cached running total, updated only on approval, kept in sync via the same race-condition-safe locked update as Contact.balance (4.7).Loss — created automatically when a batch expires (or manually for damaged/lost stock): quantity, reason (EXPIRED / DAMAGED / OTHER), lossValue = quantity × purchasePrice.Expiry AlertsA scheduled trigger calls a secured Next.js API route (POST /api/cron/expiry-check) once daily — no separate service to deploy. The route checks batches expiring within the configurable lead time (Shop.expiryAlertDays) and creates an in-app Notification, reusing the notification system already built for salary/due alerts. The endpoint is protected by a shared-secret header.Scheduling Mechanism — Final DecisionNow (pre-launch, before GCP billing is set up): cron-job.org — genuinely free, no credit card required, includes execution history/logs.Later (once on GCP billing anyway, for hosting/GCS): switch the trigger to Cloud Scheduler. Its free tier covers 3 job definitions per billing account per month — one daily job comfortably fits.Either way, the API route itself doesn't change — only who calls it. A missed run is low-stakes (a notification, not a balance/payment operation).This surfaces on the dashboard as a “Loss” section — total loss value, daily and monthly, using the same rollup approach as earnings (4.9) — sitting alongside earnings, dues, and salary payable.6.3 AI-Assisted Paper Khata MigrationProblem it solves: a shopkeeper migrating off paper isn't starting from zero — they have years of handwritten ledger pages. This feature lets them photograph old khata pages and have AI extract the data, with a mandatory human review step before anything touches the live database.EntitiesPaperImportBatch — one upload session: shopId, uploadedBy, asOfDate, status (PENDING / PROCESSING / READY_FOR_REVIEW / COMMITTED / CANCELLED).PaperImportPage — one row per uploaded photo: batchId, imageUrl (GCS), status (PENDING / EXTRACTED / FAILED / REVIEWED), rawExtractedJson, confidenceScore, resultingContactId, reviewedBy, reviewedAt.Contact.openingBalanceAsOf — seeds the balance as of the batch's chosen date rather than “now,” so the earnings rollup doesn't miscount old migrated money as today's activity.FlowShopkeeper selects one asOfDate for the whole batch, then uploads photos of paper khata pages.Each page is processed asynchronously (one Gemini API vision call per image); an FCM push notification fires when extraction is ready for review.Extracted data lands in an editable review table — nothing is written to the live database at this stage; low-confidence fields are visually flagged.Contacts with no phone or email are still allowed through, flagged as “will create as new contact”; the shopkeeper resolves duplicates visually since name alone can't be auto-deduped.On confirm, reviewed rows are fed into the existing bulk-import pipeline — same validation, same dedupe logic, no parallel system.The original page image stays attached to the resulting Contact — a durable reference if a migrated balance is ever disputed.Reviewed data is exportable as CSV independent of whether it's committed — useful as a digitized backup on its own.Scope — MVP vs. LaterMVP: extract name + final balance per contact, mapped onto the existing opening_balance field. Lower difficulty, high value, ships fast.Later: extract individual dated line items, creating backdated Transaction rows and preserving full history — deferred until MVP-tier extraction quality is validated on real samples.AI Provider — Final Decision: Gemini APIUsed via the free tier (Google AI Studio) for the accuracy spike and initial build — no credit card, generous daily limits (~1,500 requests/day on Flash models), strong multimodal support, solid Hindi/English handling out of the box. Google's free-tier terms currently exclude commercial use and permit prompts/responses to be used for model improvement — both matter once real financial data is processed at scale, so moving to a paid Gemini tier (same API, same code, billing enabled) is the plan once the feature goes live for real users.Language HandlingReal khata pages mix Hindi, English, and regional shorthand in inconsistent handwriting. Before committing to scope beyond the MVP tier, a quick spike — 2–3 real sample photos through the Gemini API's vision extraction — determines whether the review step stays light or whether extraction quality needs more work before the feature is worth shipping.7. Final Decisions on Remaining Open ItemsBulk Import CSV FormatColumns: name, phone, email, opening_balance, type (optional). name is a display label only, not a unique key. Validation requires phone or email (at least one), which serves as the dedupe key — a matching phone/email updates the existing contact rather than creating a duplicate. opening_balance must be numeric.PDF Export DesignTwo report types, both server-rendered via @react-pdf/renderer so output is consistent and the same renderer backs the shareable read-only link:Per-contact ledger — shop name/logo header, contact name + current balance, then a table of date / type / amount / running balance.All-dues summary — every contact with an outstanding due (or payable), sorted by amount.Notification Delivery ChannelWeb Push via Firebase Cloud Messaging (FCM) — free, no per-message cost, delivered through the browser's Push API, ties directly into the PWA so a shopkeeper gets a real push notification even with the app closed. In-app notifications remain as a fallback/log. SMS/WhatsApp stay deferred as a later paid addition; the Notification table includes a channel field from the start so adding them later doesn't require a schema change.Offline Write-Queue Conflict ResolutionA queued offline write goes through the same version check (4.1/4.7) as any other write once it syncs. If the version has moved on, the app does not silently merge — it surfaces a “this changed while you were offline” conflict prompt, the same visual pattern used for live-lock conflicts. One conflict-handling UI serves both cases.Half-Day Pay FractionDefaults to 50% of daily wage, stored as a per-shop setting (Shop.halfDayFraction, default 0.5) rather than hardcoded, so a shop can override it.Expiry Alert Lead TimeDefaults to 3 days, also a per-shop setting (Shop.expiryAlertDays), since some shops want a week's notice and others just a day.Stock Adjustment ApprovalA formal approval workflow, since these entries can hide shrinkage or error and deserve a second set of eyes:StockEntry gains a status: PENDING / APPROVED / REJECTED.PURCHASE and SALE entries auto-approve (routine, transactional).ADJUSTMENT and LOSS entries start PENDING.Product.currentStock only changes once an entry is APPROVED.The Owner receives a push notification (via FCM) the moment Staff submits one.Owner approves/rejects with an optional reviewNote; the decision is logged through the same audit-trail pattern used everywhere else — no new concept, just the existing pattern reapplied.8. A developer working on Inventory touches inventory/ folders across the tree and rarely opens a file another developer owns — the single biggest lever for fewer merge conflicts.Thin routes, fat services — app/api/*/route.ts files only parse the request, call a server/services/* function, and return the response. All logic lives in services/, tested independently of HTTP.Repository layer isolates Prisma — server/repositories/ is the only place prisma.* calls happen for core entities; schema changes have a one-file blast radius per entity.One validator file per resource — shared by both the API route and the client form via the same Zod schema, so server and form can never drift.Docker for parity and onboarding — docker-compose.dev.yml spins up Postgres + the app with hot reload so a new teammate is coding in minutes with no local Postgres install.Route groups (auth) / (dashboard) — organize URLs without affecting the URL path, cleanly separating authenticated vs. public layouts.Tests mirror source structure 1:1 — tests/unit/server/services/... maps to src/server/services/..., so it's always obvious where a test for a given file lives.8.2 Git Conflict Rules of ThumbNever add a new export to an existing barrel file — no barrel files are used in services/ or repositories/ for this reason; import directly from the specific file.One PR = one feature folder where possible (e.g. all of inventory/*), not scattered cross-cutting edits.Shared files everyone touches (schema.prisma, types/, config/constants.ts, tailwind.config.ts) are the few spots where conflicts happen — keep changes there small, reviewed fast, merged often.Migrations (prisma/migrations/) are append-only and generated, never hand-edited or rebased — always prisma migrate dev on top of latest main.9.
