Hisab KitabProduct Requirements DocumentDigital Khatabook / Ledger App for ShopkeepersVersion 1.0  ·  Draft for ReviewPrepared: July 20261. Overview & PurposeHisab Kitab is a digital khatabook (ledger book) application built for shopkeepers who currently track customer and vendor credit on paper. It replaces the handwritten ledger with a real-time, multi-user, auditable digital system, while preserving the simplicity shopkeepers are used to.This document defines the product scope, user flows, and functional requirements for the first shippable version (MVP) as well as the extended modules (Worker & Salary Management, Inventory & Loss Tracking, and AI-Assisted Paper Khata Migration) that are planned as part of the same build effort.This PRD is derived from three source documents already produced for this project: the Feature Spec & Design Rationale, the Application Flow document, and the Project Folder Structure document. It consolidates them into a single reference intended for both technical and non-technical stakeholders.2. Problem StatementShopkeepers who extend credit to regular customers, and who themselves buy on credit from vendors, currently track this using a handwritten paper khata (ledger book). This creates several recurring problems:Balances are not visible in real time, and there is no shared view when more than one person (e.g. an owner and a staff member) manages the same shop.Paper records are error-prone, easy to lose or damage, and offer no audit trail — it is not possible to see who changed an entry, when, or what it looked like before.There is no way to search, filter, or generate a shareable statement of dues without manually copying entries.Migrating years of handwritten history to any digital tool is a large manual effort, which is one of the biggest reasons shopkeepers delay adopting digital tools altogether.Hisab Kitab is being built to solve this with a real-time, multi-user, auditable digital ledger — plus adjacent modules (worker salary tracking, inventory/loss tracking) that round out the financial picture for a small shop.Note: The scale of this problem has not yet been quantified with market sizing or user research data (e.g. number of target shopkeepers, survey/interview evidence). This PRD proceeds without that evidence, per stakeholder direction, and the problem statement above reflects the qualitative rationale already captured in the Feature Spec document rather than measured data.3. Tech StackNext.js — full-stack React framework (single codebase for frontend and backend)PostgreSQL — relational databasePrisma — ORM for TypeScriptGCP (Google Cloud Platform) — hosting via Cloud RunGitHub Actions — CI/CD (lint, typecheck, test on every PR; build + deploy on merge to main)5. Scope5.1 In-Scope — MVP (Core Ledger)From the original problem statement:Add transactions per contact, with a race-condition-safe balance updateReal-time running balance (pushed to open screens without a refresh)Paginated transaction history (cursor-based, stays fast at any scale)Edit/delete with full audit trail (soft delete, never a hard delete)Prevention of simultaneous edits on the same transactionAdditional MVP features (in scope for v1):Role-based permissions (Owner/Staff)Due date + reminders per contactSearch & filter transactions (by contact, date range, type, amount)Receipt/photo attachment on a transactionTrash / restore for soft-deleted transactions (7-day grace period)Shareable read-only ledger link + PDF export (per-contact and all-dues summary)Multi-shop support (a user can belong to and switch between multiple shops)Analytics dashboard (money in vs out, top contacts by due, monthly trend)Opening balance / bulk CSV import for contactsIn-app notifications (edits by teammates, due-date reminders)Hindi/English language togglePWA — installable, offline-aware, with an offline write-queueDaily & monthly earnings view5.2 In-Scope — Extended ModulesPlanned as part of the same overall build, using the same underlying patterns as the core ledger (locking, audit trail, soft delete):Worker & Salary Management — attendance tracking, monthly/daily-wage pay calculation, salary paymentsInventory & Loss Tracking — products, batches, stock movements, expiry alerts, a formal approval workflow for adjustments/lossesAI-Assisted Paper Khata Migration — photograph paper ledger pages, AI-assisted extraction, mandatory human review before anything is committed5.3 Out of Scope — v1The following are explicitly deferred beyond v1. This list has been compiled from items the source documents already call out as deferred; it has not been separately reviewed or signed off by stakeholders and should be confirmed before development planning is finalized.SMS/WhatsApp notifications — deferred as a later, paid addition requiring a third-party provider (e.g. Twilio/Gupshup); push notifications via Firebase Cloud Messaging are the v1 channelDated line-item extraction for paper khata migration — v1 extracts only name + final balance per contact; extracting individual dated historical entries (to create backdated transactions) is deferred pending an accuracy spike on real samplesMigration of the expiry-alert cron trigger from a free third-party scheduler (cron-job.org) to GCP Cloud Scheduler — deferred until the project is on GCP billing anyway for other reasons; the API route itself does not changeTwo-directional real-time messaging (e.g. self-hosted Socket.io) — not needed, since all real-time requirements are server-to-browser only, covered by Server-Sent EventsPermanent/hard purge of soft-deleted records — soft delete plus a 7-day restore window satisfies the audit-trail requirement for v1; a harder-confirmed permanent purge is optional/admin-only and not committed to v1⚑ Open item: This out-of-scope list should be reviewed by the product owner and confirmed complete — there may be other features or requests (not currently documented) that should be explicitly excluded from v1 to avoid scope creep.6. System Architecture & Data Workflow (End-to-End)Hisab Kitab is a single Next.js codebase covering both frontend and backend — there are no separate frontend/backend repositories. The boundary between the two is enforced by import direction rather than by folder split: UI components never import backend logic directly; API routes are the only bridge between them.7.1 Request FlowA user action in the UI (components/) calls an API route (app/api/.../route.ts).The route parses and validates the request against a shared Zod schema (server/validators/) and calls a function in server/services/ — routes contain no business logic themselves.The service function orchestrates the use case (e.g. locking, version checks, rollup updates) and calls server/repositories/ for any database access — Prisma is only ever called from the repository layer.The response flows back through the route to the UI.7.2 Real-Time Sync (Live Balances)When a transaction is written, PostgreSQL emits a LISTEN/NOTIFY event. A server-side listener (server/realtime/pg-listener.ts) picks this up and bridges it to a Server-Sent Events (SSE) stream (app/api/sse/). Any open browser tab subscribed via the useSSE hook receives the update and re-renders the balance live — without polling and without a page refresh.7.3 Concurrency & Data IntegrityEvery transaction edit is protected by a per-contact lock (server/lib/with-contact-lock.ts) so two people cannot edit the same contact's ledger at the same instant.Transactions carry a version number; if two edits race, the second one is rejected with a conflict rather than silently overwriting the first.Deletions are soft (a deletedAt flag), never a hard row delete — this is what makes the audit trail and 7-day restore window possible.Monetary values are stored as fixed-precision decimals (never floats), to avoid rounding errors in balances.Contact balances are maintained as a rollup (a running total), recalculated defensively by a scheduled job (server/jobs/) to self-heal against any drift, rather than relying purely on incremental updates.7.4 Background JobsExpiry-check (inventory batch expiry alerts) and rollup-repair jobs run on a schedule via app/api/cron/ routes, secured by a shared secret and currently triggered by the free external scheduler cron-job.org.7.5 AI-Assisted Migration FlowA user photographs paper khata pages in-app.Images are sent to Gemini (server/lib/gemini.ts) for extraction of contact name + final balance per page.Extracted data is shown to the user for mandatory review and correction — nothing is written to the real ledger until the user explicitly confirms.On confirmation, contacts and opening balances are created through the same validated service/repository path as any other contact creation.7.6 Offline Support (PWA)The app is installable and usable offline for viewing already-synced data.Writes made offline are queued locally and flushed to the server once connectivity returns, going through the same validation and conflict-check path as an online write.8. Core Architectural Decisions & Working AssumptionsThese are the foundational decisions the technical design already rests on. They are stated here as context for anyone reading this PRD, including where the team is deliberately relying on an assumption rather than a guarantee.Locking over optimistic concurrency for transaction editsA pessimistic, per-contact lock is used instead of pure optimistic (version-only) concurrency, because a shop's transaction volume is low enough that lock contention is rare, and the simplicity of 'you can't even open the edit form while someone else is in it' is easier for non-technical users to understand than a conflict-resolution screen.Server-Sent Events over WebSocketsAll real-time requirements in this product are one-directional (server pushes an updated balance to the browser); no client-to-client messaging is needed. SSE is simpler to operate on Cloud Run than a persistent bidirectional WebSocket connection, and is sufficient for this use case.Soft delete, not hard deleteEvery deletable record (transactions, etc.) is soft-deleted with a 7-day restore window, satisfying the audit-trail requirement without needing a separate archive system.Fixed-precision decimal moneyAll monetary fields use fixed-precision decimals rather than floating-point numbers, to avoid rounding errors accumulating across thousands of transactions.Rollup balances with self-healing repairContact balances are stored as a running rollup for fast reads, with a background job that recalculates from the transaction history to correct any drift — this assumes the rollup-repair job runs reliably; if the underlying scheduler is delayed or fails, balances could drift until the next successful run.Free-tier dependencies (working assumptions, not guarantees)cron-job.org (free) is used to trigger scheduled jobs in v1, with a documented plan to migrate to GCP Cloud Scheduler later. This assumes cron-job.org's reliability is acceptable for expiry-alert and rollup-repair timing in v1; it is not a guaranteed-uptime service.Gemini's free tier is used for paper khata extraction. This assumes free-tier rate limits and accuracy are sufficient for expected v1 usage volumes; both should be revisited if usage grows or extraction quality is inconsistent.Dataset availability for AI migration⚑ Open item: Sample paper-khata images (or any real transaction data) for testing the AI extraction flow have not yet been confirmed available with a data/engineering team. This should be resolved before the extraction accuracy work begins, since extraction quality cannot be properly validated without representative samples.9. Extended Module Specifications9.1 Worker & Salary ManagementTracks shop staff separately from ledger contacts, covering attendance and pay.Add a worker with either a fixed monthly salary or a daily wage rate.Mark daily attendance: present, absent, half-day, or leave.Pending salary is calculated from the attendance record and pay type, shown as 'Salary Payable' on the dashboard.Record salary payments, which reduce the payable amount and are themselves auditable (same soft-delete/edit-trail approach as ledger transactions).9.2 Inventory & Loss TrackingTracks stock at the product/batch level, tied to purchases and sales, with a deliberate approval step for anything that reduces stock unexpectedly.Products and stock batches carry quantity and expiry date.Purchases and sales automatically move stock in the corresponding direction.Any adjustment or loss entry submitted by Staff requires Owner approval before it affects official stock counts — this is a deliberate control point to catch errors or shrinkage before they're silently absorbed into the numbers.A scheduled job checks for upcoming batch expiries and raises an in-app notification ahead of time.9.3 AI-Assisted Paper Khata MigrationAimed at the biggest adoption barrier for paper-ledger shopkeepers: migrating existing history without a large manual data-entry effort.v1 scope: extract contact name and final balance per page — sufficient to seed opening balances for existing contacts.Every extraction is shown to the user for review and correction before anything is committed to the real ledger; nothing is auto-committed.Users can export the reviewed extraction as a CSV even without committing it to their live ledger, as a standalone digitization benefit.Dated line-item extraction (recreating each individual historical transaction with its original date) is deferred beyond v1 — see Section 5.3, Out of Scope.10. Appendix — Project Folder StructureReference structure for engineering; included here for traceability between this PRD and the codebase, per the Project Folder Structure document.Guiding principlesFrontend/backend separation lives inside a single src/ tree (not two repos), enforced by import direction: components/ never imports server/ directly — API routes are the only bridge.Folders below the first level are organized by feature/domain (contacts, transactions, workers, inventory, paper-import), not by technical type — matching the entities in this PRD so a developer working on one feature mostly stays within one set of folders.Thin routes, fat services: API route files only parse the request, call a service function, and return the response. All logic lives in server/services/, testable independently of HTTP.server/repositories/ is the only layer that calls Prisma directly for core entities, so schema changes have a contained blast radius.One Zod validator file per resource, shared by both the API route and the client-side form, so server and client validation never drift apart.Docker Compose provides a one-command local setup (Postgres + app with hot reload) for onboarding, and a prod-shaped Compose file that CI builds against to catch environment drift early.Tests mirror the source tree 1:1, so it's always obvious where a test for a given file lives.Top-level layoutsrc/app/ — Next.js routes only (route groups for auth vs. authenticated dashboard; app/api/ for backend endpoints, including sse/ and cron/)src/components/ — presentation only, split into ui/ (dumb primitives), layout/ (shell/nav), and features/ (feature-scoped components)src/server/ — all business logic: services/, repositories/, validators/, middleware/, realtime/, jobs/, lib/src/hooks/, src/lib/, src/i18n/, src/config/, src/types/ — shared client-safe utilities, translations, feature flags, and shared typesprisma/ — schema, migrations, seed scriptdocker/, infra/, .github/workflows/ — containerization, GCP/cron infra notes, and CI/CD pipelinestests/ — unit, integration (real Postgres via Docker), and e2e (Playwright), mirroring src/ structure
