Why This Shape (Monorepo Design Decisions)

1. Single repo, independent deployments.
   Frontend (apps/web) and backend (apps/server) live in ONE Git repository but are built
   and deployed as SEPARATE containers/services. The web app hits the API server via
   NEXT_PUBLIC_API_URL — no shared runtime, no coupled deploys.

2. Shared packages eliminate code drift.
   packages/shared/ contains Zod validators and TypeScript types used by BOTH apps.
   When a schema changes, the compiler catches mismatches in both web and server at build
   time. No version of "the form sends X but the API expects Y" ever ships.

3. Feature-based, not type-based, past the first level.
   Inside components/features/, routes/, and services/, folders are named after domain
   concepts (contacts, transactions, workers, inventory, paper-import) — matching the
   entities in the PRD. A dev working on Inventory touches inventory/ folders across the
   tree and almost never opens a file another dev owns. This is the single biggest lever
   for fewer merge conflicts.

4. Thin routes, fat services.
   routes/*.ts files only do: parse request → call a services/* function → return response.
   All logic (locking, version checks, rollup updates) lives in services/, tested
   independently of HTTP. Keeps route files small and mergeable.

5. Repository layer isolates Prisma.
   apps/server/src/repositories/ is the ONLY place prisma.* calls happen for core entities.
   If the schema changes, the blast radius is one file per entity, not scattered across
   services. The Prisma client itself lives in packages/database/ — a single source of truth.

6. One validator file per resource.
   packages/shared/src/validators/transaction-schema.ts is shared by both the API route
   (request validation) and the client form (same Zod schema) — zero drift.

7. Docker for parity + onboarding.
   docker/web/ and docker/server/ have separate Dockerfiles. A new teammate runs
   `docker compose up` and gets the full stack (frontend + API + Postgres) running with
   hot reload in minutes — no local Postgres install needed.

8. Turborepo for build orchestration.
   turbo.json defines the build pipeline: packages first, then apps. `turbo run build`
   only rebuilds what changed. `turbo run dev` starts both apps in parallel.

9. Tests mirror apps structure.
   tests/web/ and tests/server/ map to apps/web/ and apps/server/ respectively. It's
   always obvious where a test for a given file lives, and two people adding tests for
   different features don't collide.

Git Conflict Rules of Thumb for the Team

- Never add a new export to an existing "barrel" file — import directly from the specific file.
- One PR = one feature folder where possible (e.g. all of inventory/*).
- Shared files everyone touches (schema.prisma, packages/shared/src/types/, config/constants.ts)
  are the few spots where conflicts will happen — keep changes small, reviewed fast, merged often.
- Migrations (packages/database/prisma/migrations/) are append-only and generated, never
  hand-edited or rebased — always `prisma migrate dev` on top of latest main.
- Web-only PRs should NOT trigger server rebuilds (Turborepo handles this automatically).


Getting Started

  cp .env.example .env
  pnpm install                                    # installs all workspace dependencies
  docker compose -f docker/docker-compose.dev.yml up -d   # postgres
  pnpm --filter @hisab-kitab/database db:migrate  # run migrations
  pnpm --filter @hisab-kitab/database db:seed     # seed data
  pnpm dev                                        # starts both web (3000) + server (4000)
