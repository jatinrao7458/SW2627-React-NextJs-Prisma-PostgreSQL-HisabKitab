Hisab KitabImplementation PlanPhase-Based Build Roadmap — MVP Through Extended ModulesVersion 1.0  ·  Draft for ReviewPrepared: July 20261. Overview & ApproachThis plan sequences the build of Hisab Kitab into phases, from infrastructure setup through the MVP core ledger, then the three extended modules, then hardening and launch. It is written generically — without assuming a specific team size — so tasks can be picked up by one developer working sequentially or split across a small team working phases in parallel where noted.It builds directly on three prior deliverables: the PRD (Hisab_Kitab_PRD.docx), the backend schema (schema.prisma / Hisab_Kitab_Backend_Schema.docx), and the folder structure already defined in the source documents. This plan does not repeat feature detail already captured there — it sequences the work and defines what "done" looks like at each stage.1.1 Guiding PrinciplesInfrastructure and auth first — every feature phase depends on a working database, a deployable app, and real sessions; building these last would mean retrofitting identity into locking, audit trails, and permissions after the fact.Core ledger before anything else — the extended modules deliberately reuse the ledger's patterns (locking, soft delete, audit trail); building them first would mean inventing those patterns twice.Thin vertical slices, not horizontal layers — each phase ships a usable, testable piece of the product end-to-end (UI + API + DB), rather than "build all the database, then all the API, then all the UI."Extended modules build sequentially after the MVP core ships. This is an assumption, not a requirement stated in the source docs — flagged in Section 7, Open Items — but it matches how the Feature Spec already frames the extended modules as reusing patterns "already built for the core ledger."Each phase ends with an explicit exit criteria checklist — a phase is not "done" because code was written, but because it was tested and meets the criteria.2. Phase 0 — Foundation & InfrastructureGoal: a deployable skeleton app with database, auth, and CI/CD working end-to-end, before any real feature is built.2.1 TasksInitialize the Next.js project using the folder structure already defined (src/app, src/components, src/server, prisma/, docker/, infra/, tests/).Set up docker-compose.dev.yml (Postgres + app, hot reload) and docker-compose.yml (prod-shaped, used by CI).Write prisma/schema.prisma (already produced — see schema.prisma) and run the first migration against local Docker Postgres.Write prisma/seed.ts with a minimal seed: one demo Shop, one Owner User, a couple of Contacts and Transactions — enough to develop against without manual setup every time.Implement NextAuth with the Prisma adapter (Account/Session/VerificationToken already in the schema); ship email/password registration + login first.Implement server/middleware/require-session.ts and a stub server/middleware/require-role.ts (full rule table comes in Phase 1, but the enforcement point should exist from day one).Set up .github/workflows/ci.yml (lint, typecheck, test on every PR) and deploy.yml (build image, push, deploy to Cloud Run on merge to main).Provision the GCP Cloud Run service and a managed Postgres instance (or Cloud SQL), and confirm a successful first deploy of the skeleton app.2.2 Exit CriteriaA developer can clone the repo, run the Getting Started steps from the Folder Structure doc, and be coding in minutes with zero manual Postgres setup.A user can register, log in, and land on an empty dashboard shell, in both local dev and the deployed Cloud Run environment.CI runs lint/typecheck/test on every PR; a merge to main deploys automatically.3. Phase 1 — MVP Core LedgerGoal: the heart of the product — contacts, transactions, real-time balances, and the safety mechanisms (locking, audit trail, race-condition handling) that everything else depends on.3.1 Tasks — in dependency order3.1.1 Shop & MembershipShop model + Shop onboarding flow (create shop, seed default settings: halfDayFraction, expiryAlertDays).ShopMember join table + role assignment (Owner on creation).Shop switcher UI + "last-used shop" preference for users with 2+ shops.Implement the full require-role.ts permission matrix (Section 4.2 of the schema doc) — this gates every subsequent feature, so it belongs here, not bolted on later.⚑ Open item: Resolve the last-Owner-leaving-a-shop edge case (see PROJECT_CONTEXT.md §5) before shipping ShopMember removal/role-change UI.3.1.2 ContactsAdd/edit/list Contact, with opening balance.Search & filter (name/phone/email, debounced).3.1.3 Transactions — the core write pathAdd transaction (You Gave / You Got) with the row-locked balance update (SELECT ... FOR UPDATE) — build and load-test this before anything else touches it, since every other feature depends on this being correct.Cursor-based pagination on transaction history, keyed on (createdAt, id), with balanceAfter cached per row.Edit/delete with TransactionAudit (old/new JSON snapshot, written in the same DB transaction as the change).Hybrid concurrency: lockedBy/lockedAt soft-lock with heartbeat + TTL, plus the version safety net underneath.Trash / Restore screen (7-day grace period), gated by the staffCanRestoreTrash setting.3.1.4 Real-Time SyncPostgres LISTEN/NOTIFY → SSE bridge (server/realtime/pg-listener.ts, app/api/sse/).useSSE client hook; live balance updates on any open contact ledger.Live lock-status broadcast ("being edited by Priya") over the same channel.3.1.5 Dashboard & RollupsEarningsRollup table + atomic update inside every write that affects it.Dashboard: Total Dues, Total Payable, daily/monthly earnings, top contacts by due, money in/out trend.3.1.6 Supporting MVP FeaturesDue date + reminder notifications per contact.Receipt/photo Attachment (polymorphic table) on transactions.Shareable read-only ledger link + PDF export (per-contact, all-dues summary).Bulk CSV import for contacts (validated against the documented column format: name, phone, email, opening_balance, type).In-app + push notifications (FCM Web Push), with the Notification.channel field ready for SMS/WhatsApp later.Hindi/English i18n toggle.PWA shell: installable, offline viewing, offline write-queue with the same version-check conflict resolution as online writes.3.2 Exit CriteriaTwo browser sessions on the same contact show a live balance update within ~1 second of a transaction being added, with no manual refresh.Two simultaneous edits to the same transaction are correctly blocked (lock) or rejected (stale version) — verified with a deliberate concurrency test, not just manual testing.A shop with several thousand seeded transactions still loads a transaction history page and the dashboard in well under a second.Every delete is soft; a deleted transaction can be restored within 7 days and its audit trail is intact.The app is installable as a PWA and remains usable (read-only) with no network connection.4. Phase 2 — Extended Module: Worker & Salary ManagementGoal: reuse the ledger's own patterns (locking, audit trail, soft delete) for a second domain, proving those patterns generalize before Inventory adds more complexity.4.1 TasksWorker model (payType MONTHLY/DAILY_WAGE, monthlySalary/dailyWageRate, designation, joiningDate, active).Attendance: one row per worker per day (PRESENT/ABSENT/HALF_DAY/LEAVE), unique per (workerId, date).Pending salary calculation: monthly = monthlySalary × full months − total paid; daily wage = (PRESENT days × rate) + (HALF_DAY days × rate × halfDayFraction) − total paid.SalaryPayment with the same version/lockedBy/lockedAt pattern as Transaction, plus SalaryAudit mirroring TransactionAudit.Salary payment approval: Owner always; Staff record-only per the permissions matrix."Salary Payable" figure surfaced on the dashboard alongside dues/payables.4.2 Exit CriteriaMarking attendance and recording a salary payment for the same worker from two sessions at once behaves correctly under the existing concurrency pattern — no separate bug class introduced.Pending salary for both pay types matches a hand-calculated example exactly, including a half-day-affected month.Dashboard's Salary Payable figure updates immediately after a payment is recorded.5. Phase 3 — Extended Module: Inventory & Loss TrackingGoal: add stock tracking with a deliberate approval workflow, and the scheduled expiry-alert job — the first cron-triggered feature in the product.5.1 TasksProduct, ProductBatch (batch-level expiry/quantity/purchase price).StockEntry (PURCHASE/SALE/ADJUSTMENT/LOSS) with the approval workflow: PURCHASE/SALE auto-approve; ADJUSTMENT/LOSS start PENDING and require Owner approval before Product.currentStock changes.Product.currentStock kept in sync using the same row-locked update pattern as Contact.balance.Loss records (EXPIRED/DAMAGED/OTHER), auto-created on batch expiry or manually for damage.POST /api/cron/expiry-check route, secured by a shared-secret header, checking batches expiring within Shop.expiryAlertDays.Register the route with cron-job.org (the documented v1 scheduler decision) and confirm execution history/logs are visible.Owner push notification the moment Staff submits a PENDING stock entry; approve/reject with an optional reviewNote, logged through the same audit pattern."Loss" section on the dashboard (daily/monthly, same aggregation approach as earnings).5.2 Exit CriteriaA Staff-submitted ADJUSTMENT or LOSS entry does not affect currentStock until an Owner explicitly approves it, verified end-to-end.The expiry-check cron job correctly identifies a seeded batch expiring within the alert window and creates exactly one notification per batch (no duplicates on repeated runs).Dashboard Loss figures match the sum of approved Loss records for the period, exactly.6. Phase 4 — Extended Module: AI-Assisted Paper Khata MigrationGoal: let a shopkeeper digitize existing paper records with AI assistance, with a mandatory human review step — built last since it's the least core to daily operation and depends on a dataset that is not yet confirmed available.⚑ Open item: Do not start the extraction-accuracy work until sample paper-khata images (or real transaction data) are confirmed available — see PROJECT_CONTEXT.md §5, item 3. This phase should begin with that confirmation, not with code.6.1 TasksAccuracy spike first: run 2–3 real sample photos through the Gemini API's vision extraction and evaluate accuracy before committing further engineering time — this determines whether MVP-tier scope (name + final balance only) is viable as specified.PaperImportBatch (one upload session, single asOfDate for the whole batch) and PaperImportPage (per-photo status, rawExtractedJson, confidenceScore).Async processing: one Gemini vision call per image, no blocking UI; progress screen + push notification when READY_FOR_REVIEW.Editable review table — nothing written to the live database until explicit confirmation; low-confidence fields visually flagged.Rows with no phone/email allowed through, explicitly flagged as "will create as new contact"; duplicate resolution is manual/visual.On confirm, feed reviewed rows into the existing bulk-import pipeline (same validation/dedupe logic as CSV import) — not a parallel import system.Attach the original page image to the resulting Contact via the existing Attachment pattern.CSV export of reviewed data, independent of whether it's committed to the app.Contact.openingBalanceAsOf set to the batch's asOfDate so the earnings rollup doesn't miscount migrated money as new activity.6.2 Exit CriteriaThe accuracy spike result is documented and reviewed before proceeding past MVP-tier scope (name + balance only).A committed batch produces Contacts identical in structure to ones created via manual entry or CSV import — no separate code path with different validation.A shopkeeper can export a reviewed batch as CSV without ever committing it, and that CSV matches the documented bulk-import column format.7. Phase 5 — Hardening & Launch ReadinessGoal: everything works individually; this phase confirms it works together, under load, and safely, before real shopkeepers depend on it.7.1 TasksLoad test the transaction write path and dashboard reads at a realistic multi-shop, multi-thousand-transaction scale — confirm cursor pagination and rollup reads stay flat as data grows.Concurrency test suite: simultaneous edits on the same transaction/salary payment, simultaneous new transactions on the same contact, offline-queue writes syncing into a since-changed record — all three conflict paths from the docs, each verified with an automated test, not just manual clicking.Security pass: confirm every server/services/* function is reachable only through require-session.ts + require-role.ts, confirm the cron endpoint's shared-secret header is actually enforced, confirm shared read-only ledger links can't be escalated into write access.Full permissions matrix regression test (Section 4.2 of the schema doc) — every row of the Owner/Staff table exercised as an automated test, not just spot-checked.Backup/restore drill on the production Postgres instance — confirm a restore actually works before launch, not just that backups are being taken.Monitoring & alerting: Cloud Run request latency/error rate, cron job failure alerts (the expiry-check job going silent should be noticed), SSE connection health.Accessibility and low-end-device pass — target users are on phones at a shop counter, often mid-range Android devices on patchy network; test on that profile, not just a developer laptop.7.2 Exit CriteriaAll three documented conflict-handling paths (live lock, stale version, offline sync) pass automated tests, not just manual verification.A simulated Cloud Run outage/restart does not lose any queued offline writes or leave a transaction in a permanently locked state (TTL release confirmed).A backup taken today can be restored to a working database today — tested, not assumed.8. Cross-Cutting Practices (apply throughout every phase)8.1 Testing StrategyLayerWhat it coversLocation (per Folder Structure doc)Unitserver/services/* logic in isolation — salary math, stock approval rules, balance calculationstests/unit/ (mirrors src/server/)IntegrationReal Postgres via Docker — row-locking, concurrency, rollup consistencytests/integration/End-to-endFull user flows through the UItests/e2e/ (Playwright, mirrors app/ routes)Concurrency-sensitive logic (locking, race-condition-safe balance updates, approval workflows) should always have an integration test that exercises real simultaneous writes against real Postgres — a unit test with mocked data cannot catch a row-locking bug.8.2 CI/CDEvery PR: lint, typecheck, unit + integration tests (ci.yml). A PR should not be mergeable if any of these fail.Merge to main: build image, push, deploy to Cloud Run (deploy.yml).Migrations are append-only and generated (prisma migrate dev), never hand-edited or rebased — always applied on top of latest main, per the Folder Structure doc's git rules.8.3 EnvironmentsLocal: docker-compose.dev.yml — Postgres + app with hot reload, one command to start.CI: docker-compose.yml (prod-shaped) — catches environment drift before it reaches production.Production: Cloud Run + managed Postgres on GCP.8.4 Git & CollaborationOne PR = one feature folder where possible (e.g. all of inventory/*), not scattered cross-cutting edits — per the Folder Structure doc's conflict-reduction rules.Never add a new export to a barrel file — import directly from the specific file (no barrel files in services/ or repositories/).Shared files everyone touches (schema.prisma, types/, config/constants.ts) get small, fast-reviewed, frequently-merged changes — never left in a long-lived branch.9. If parallel development across a team is actually intended, Phases 2–4 can run concurrently with each other (they don't depend on one another) but should still start after Phase 1's core ledger and permission system are stable, since all three reuse those patterns directly.⚑ Open item: Schema has not been run through the actual Prisma CLI (prisma validate / prisma format) due to no network access in the environment where it was drafted — run this as the very first Phase 0 task, before writing the first migration.
